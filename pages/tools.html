프로그램별 상세 가이드

각 프로그램을 실제로 어떻게 사용하면 될까요?

이 챕터는 프로그램별 실제 진행 순서를 확인하는 곳입니다. 제작 흐름을 먼저 보고 왔다면, 지금 필요한 목적별 버튼이나 프로그램별 버튼을 눌러 세부 방법을 펼쳐보세요. 다만 이 설명은 프로그램의 가능성을 닫아두기 위한 목록이 아니라, 선생님의 수업 아이디어를 확장하기 위한 출발점입니다.

프로그램별 상세 가이드

이 영역은 실제 프로그램 사용법입니다. 먼저 수업 기능을 고르고, 필요하면 웹 배포·웹앱 확장·데이터 저장 같은 운영 기반을 붙여서 확인하세요.

선택 방법 1차 선택에서는 “학생이 무엇을 하게 할지”를 고릅니다. 2차 선택에서는 그 기능에 맞는 프로그램 하나를 고릅니다. 비교가 필요할 때만 `선택 분야 전체 보기`를 누르세요.

수업 기능 안내 페이지 설명을 보고 있습니다

안내 페이지를 만들고 싶다면 Canva AI/웹페이지, Google Sites처럼 진입 장벽이 낮은 도구부터 확인합니다.

프로그램의 가능성은 수업 아이디어에 따라 달라집니다 이 페이지의 분류는 “이 도구는 이것밖에 못 한다”는 뜻이 아닙니다. 예를 들어 Google Sites는 단순 자료실로 끝날 수도 있지만, Forms, Sheets, Apps Script를 연결하면 제출 확인 허브가 될 수 있습니다. Canva도 안내 디자인에서 시작해 간단한 상호작용 활동과 데이터 시각화로 확장될 수 있습니다. 중요한 것은 도구의 이름보다 수업에서 학생이 어떤 경험을 하게 할지 먼저 정하는 것입니다.

이 상세 가이드를 읽는 순서

프로그램별 설명이 길게 느껴진다면 아래 네 가지 질문만 먼저 확인하세요. 이 질문에 답하면 어떤 도구를 열어야 하는지, 어디서 막힐지, AI에게 무엇을 물어야 하는지가 보입니다.

초보자 추천 읽기법 처음부터 모든 프로그램 설명을 끝까지 읽지 않아도 됩니다. 1차 선택에서 필요한 기능을 고르고, 2차 선택에서 프로그램 하나를 고른 뒤, 각 카드의 `3분 요약`만 먼저 보세요. 그다음 실제로 만들기로 결정한 도구만 `실제 진행 순서`와 `막혔을 때 추가로 물어볼 말`을 펼쳐보면 됩니다.
1. 학생이 무엇을 하게 되나요?

읽기, 누르기, 제출하기, 다시 확인하기 중 하나로 먼저 나눕니다. 읽기만 하면 안내 도구, 누르면 활동 도구, 제출하면 응답 도구가 필요합니다.

2. 파일을 직접 만지나요?

Canva, Google Sites, Genially는 화면 안에서 만들고 공유할 수 있습니다. AI가 만든 `index.html`은 Netlify, GitHub Pages, Cloudflare Pages 같은 배포 도구가 필요합니다.

3. 학생 응답을 저장하나요?

단순 제출은 Forms나 Tally가 쉽습니다. 계속 확인할 현황은 AppSheet를 검토합니다. 직접 만든 웹앱의 데이터 저장은 Supabase나 Firebase가 필요할 수 있습니다.

4. 공개해도 안전한가요?

학생 이름, 얼굴 사진, 학번, 성적, 내부 문서 링크가 들어가면 배포보다 먼저 공개 범위와 권한을 확인합니다.

Canva AI첫 성공 기준은 “학생이 한 화면에서 활동 순서와 버튼을 이해하는가”입니다.교사용 표 정리와 차트는 Canva Sheets로 시작할 수 있습니다.학생 제출을 안전하게 모아야 하면 Forms, Tally, AppSheet와 비교합니다.
Google Sites첫 성공 기준은 “학생 계정에서 자료와 설문이 모두 열리는가”입니다.사이트 공개와 Drive 파일 공개는 따로 확인해야 합니다.
Genially첫 성공 기준은 “모든 버튼, 팝업, 퀴즈 이동이 작동하는가”입니다.점수나 제출 기록은 외부 폼과 연결하는 편이 안전합니다.
Tally·Forms첫 성공 기준은 “학생이 제출하고 교사가 응답을 확인할 수 있는가”입니다.이름을 꼭 받아야 하는지부터 줄여봅니다.
AppSheet첫 성공 기준은 “시트의 열 이름과 앱 화면이 교사의 관리 목적에 맞는가”입니다.처음에는 교사용 내부 앱으로 테스트합니다.
Netlify첫 성공 기준은 “`index.html`이 들어 있는 폴더를 올렸을 때 주소가 생기고 이미지까지 보이는가”입니다.파일 하나만 올리면 이미지가 빠질 수 있습니다.
GitHub Pages첫 성공 기준은 “저장소에 `index.html`을 올리고 Pages에서 main 또는 master 브랜치를 선택했는가”입니다.저장소를 온라인 자료함으로 이해하면 쉽습니다.
Cloudflare Pages첫 성공 기준은 “GitHub 저장소와 연결해 자동 배포가 되는가”입니다.GitHub가 아직 낯설면 Netlify로 먼저 성공 경험을 만드는 것이 낫습니다.
Supabase·Firebase첫 성공 기준은 “익명·비민감 테스트 데이터가 저장되고 보안 규칙을 설명할 수 있는가”입니다.실제 학생 개인정보는 학교 정책 확인 후 진행합니다.
고급 도구는 필수 단계가 아닙니다 Supabase, Firebase, Cloudflare Workers, Vercel API, Apps Script 자동화는 강력합니다. 하지만 처음부터 반드시 써야 하는 도구는 아닙니다. 단순 안내는 Canva AI, Google Sites, Notion으로도 충분합니다. 단순 제출은 Google Forms나 Tally가 더 안전할 수 있습니다. 학생 기록 저장, 로그인, 실시간 동기화, 자동 배포가 정말 필요할 때만 확장 단계로 검토하세요.

마크다운 검토 포인트: 단순 조작보다 개념 정확성, 그래프 판별 포인트, 환경 문제 연결이 중요합니다.

응답·분석

마크다운 검토 포인트: 학생 공개용이 아니라 교사용 수업 설계·모둠 구성·관찰 기록 자료로 다루는 것이 안전합니다.

사례 보기 탭에서는 이 화면들을 더 길게 분석합니다. 이 페이지에서는 “어떤 기능을 어떤 도구로 만들 수 있는지”를 파악하는 정도로만 사용합니다.

수업용 웹페이지를 이루는 핵심 요소

웹페이지 제작은 프로그램 이름부터 고르는 일이 아닙니다. 안내, 조작, 응답, 배포, 확장, 저장 중 어떤 요소가 필요한지 정하면 도구 선택이 훨씬 쉬워집니다.

수업 기능

1. 안내 페이지

학생이 “무엇을 언제 어떻게 해야 하는지” 확인하는 기본 화면입니다. 수업 안내판, 수행평가 설명서, 행사 안내, 학부모 안내처럼 읽고 따라가면 되는 자료에 적합합니다.

  • 좋은 구성: 제목, 대상, 기간, 준비물, 활동 순서, 제출 링크, 문의 방법
  • 추천 도구: Canva AI/Canva 웹페이지, Google Sites
  • 판단 질문: 학생이 이 주소 하나만 보고 다음 행동을 할 수 있는가?
수업 기능

2. 상호작용 활동

학생이 버튼을 누르고, 답을 고르고, 카드를 뒤집고, 지도나 이미지를 탐색하는 활동입니다. 단순히 읽는 자료보다 조작 과정 자체가 학습이 될 때 필요합니다.

  • 가능한 결과물: OX 퀴즈, 플래시카드, 매칭 게임, 팝업 설명창, 지도 탐색, 계산기, 카운트다운
  • 추천 도구: Canva AI의 코드 생성 방식, Genially, HTML + JavaScript
  • 판단 질문: 학생이 누르는 행동이 학습 목표와 직접 연결되는가?
수업 기능

3. 응답·관리

학생의 신청, 피드백, 활동 완료, 모둠 상태, 제출 여부를 모으고 확인하는 요소입니다. 웹페이지가 단순 안내문을 넘어 수업 운영 도구가 되는 지점입니다.

  • 쉬운 수집: Google Forms, Tally
  • 현황 관리: Google Sheets, AppSheet, Canva Sheets
  • 판단 질문: 단순히 제출받으면 되는가, 아니면 교사가 계속 확인하고 관리해야 하는가?
운영 기반

4. 웹 배포

내 컴퓨터 안의 HTML 파일을 학생이 주소로 열 수 있게 만드는 과정입니다. Canva, Google Sites처럼 자체 게시 기능이 있는 도구는 배포가 간단합니다. 직접 만든 HTML은 별도 배포 도구가 필요합니다.

  • 빠른 첫 배포: Netlify
  • 수정 기록과 장기 관리: GitHub Pages
  • 자동 배포와 운영 안정성: Cloudflare Pages
운영 기반

5. 웹앱 확장

여러 화면, 필터, 검색, 점수 계산, 진행 상태 표시처럼 화면이 계속 바뀌는 구조입니다. 이 단계부터 HTML, CSS, JavaScript의 역할을 AI와 함께 점검하는 것이 좋습니다.

  • HTML: 화면의 뼈대와 글자
  • CSS: 색, 간격, 카드, 모바일 반응형
  • JavaScript: 버튼 반응, 퀴즈 채점, 필터, 데이터 처리
운영 기반

6. 데이터 저장

웹페이지가 학생 응답이나 활동 기록을 “기억”하게 만드는 요소입니다. 이 단계에서는 API, 백엔드, 데이터베이스, 보안 규칙을 함께 봐야 합니다.

  • 가벼운 백엔드: Apps Script, AppSheet
  • 입력 수집형 저장: Google Forms, Tally
  • 표처럼 저장하고 조회: Supabase
  • 실시간 동기화·로그인·파일 저장: Firebase
  • 판단 질문: 학생 개인정보를 꼭 저장해야 하는가, 익명이나 모둠 단위로 줄일 수 있는가?
저장·백엔드 기능 이해하기 학생 응답 저장, 조회, 자동 처리의 가능 범위와 한계를 봅니다.

백엔드 기능은 어디까지 가능할까요?

초보자에게 백엔드는 “학생이 보는 화면 뒤에서 저장·조회·확인·전송을 처리하는 역할”로 설명하면 충분합니다. 다만 모든 도구가 같은 수준의 백엔드는 아니므로, 가능 범위와 한계를 함께 봐야 합니다.

입력 수집형

Google Forms와 Tally는 학생 응답을 받아 저장해주는 역할을 합니다. 교사는 폼 링크만 연결하면 됩니다. 다만 직접 만든 웹앱처럼 화면과 저장 흐름을 자유롭게 통제하기는 어렵습니다.

Google 기반 가벼운 백엔드

Apps Script와 AppSheet는 Google Sheets를 중심으로 작동합니다. 저장, 조회, 자동 이메일, 제출 현황 확인 같은 기능을 만들 수 있습니다. 학교 Google 계정 환경과 잘 맞습니다. 다만 권한과 실행 제한을 확인해야 합니다.

개발형 백엔드

Supabase와 Firebase는 직접 만든 웹앱에 고급 기능을 붙일 때 씁니다. 예: 로그인, 데이터베이스, 저장소, 실시간 동기화 강력하지만 보안 규칙과 개인정보 설계가 중요합니다.

배포 플랫폼의 함수 기능

Google Sites로 할 수 있는 것

수업 자료실

Docs, Slides, Sheets, Forms, Calendar, Maps, Drive 파일을 삽입해 학교 계정 기반 자료 흐름을 만들 수 있습니다.

Apps Script 확장

제출 상태 조회, 신청 결과 확인, 자동 이메일 안내처럼 Google Sheets 기반 자동화 화면을 붙일 수 있습니다.

공개 범위 관리

전체 공개, 링크 공개, 학교 계정 제한을 선택할 수 있지만, 삽입된 Drive 파일 권한도 따로 확인해야 합니다.

한계

화면 디자인 자유도와 게임형 상호작용은 제한적입니다. 조작형 활동은 Genially나 HTML/JavaScript가 더 적합합니다.

실제 진행 순서

  1. Google Drive에 수업 자료 폴더를 먼저 만듭니다. 문서, 슬라이드, 활동지 PDF, 설문 링크를 한곳에 모아둡니다.
  2. Google Sites에서 새 사이트를 만들고 사이트 제목을 정합니다. 제목은 학생이 바로 알아볼 수 있게 씁니다.
  3. 첫 화면에는 “이 사이트에서 무엇을 하면 되는지”를 3~5줄로 짧게 안내합니다.
  4. 상단 메뉴를 만듭니다. 예: 수업 안내, 활동 자료, 제출 링크, 평가 기준, 자주 묻는 질문
  5. 각 페이지에는 한 가지 목적만 담습니다. 자료실 페이지와 제출 페이지를 섞지 않는 것이 좋습니다.
  6. 삽입 메뉴에서 Docs, Slides, Sheets, Forms, Calendar, Maps, Drive 파일을 필요한 위치에 추가합니다.
  7. 학생이 제출해야 하는 활동은 Google Forms 버튼이나 Classroom 과제 링크로 연결합니다.
  8. 활동 순서가 중요한 경우 번호를 붙여 `1. 자료 보기 → 2. 활동지 작성 → 3. 제출하기`처럼 배치합니다.
  9. 게시 전에 삽입한 Drive 파일의 공유 권한을 하나씩 확인합니다. 사이트 공개와 파일 공개는 별도입니다.
  10. 학생 계정으로 열어 보고 문서, 영상, 설문, 지도 링크가 모두 열리는지 확인합니다.
  11. 게시 주소를 정하고 학교 내부 공개, 링크 공개, 전체 공개 중 하나를 선택합니다.
  12. 학급방에 공유하기 전 모바일 화면에서 메뉴가 너무 많지 않은지 확인합니다.

한계

전문 백엔드처럼 복잡한 로그인, 대규모 동시 접속, 세밀한 데이터 권한 관리를 맡기기에는 한계가 있습니다. 실행 시간과 호출 횟수 제한도 고려해야 합니다.

Google Sheets를 중심으로 신청 현황, 제출 확인, 간단 조회, 자동 안내 정도를 만들 때 가장 현실적입니다.

Docs, Slides, Sheets, Forms, Drive 파일을 삽입해 활동지, 발표 자료, 설문, 제출 링크를 한곳에 모읍니다.

사이트 공개 범위와 삽입한 Drive 파일 공개 범위를 각각 확인합니다. 사이트만 공개하고 파일이 잠겨 있으면 학생이 열 수 없습니다.

Apps Script 연결 실제 순서

  1. Google Sheets나 Forms로 학생 입력을 받을 기본 자료를 준비합니다.
  2. 시트에 실제 학생 정보 대신 테스트용 예시 데이터 2~3개를 먼저 넣습니다.
  3. Apps Script에서 필요한 자동화 코드를 만듭니다. 예: 제출 확인, 시트 조회, 이메일 안내
  4. Apps Script를 웹앱으로 배포할 때 실행 권한과 접속 가능 대상을 확인합니다.
  5. 접속 URL을 복사합니다.
  6. Google Sites에서 삽입 또는 Embed 메뉴를 열고 웹앱 URL을 붙여넣습니다.
  7. 학생 계정, 교사 계정, 비로그인 상태에서 각각 열리는지 확인합니다.

Google Sites의 강점은 “모든 것을 직접 코딩하지 않아도 된다”는 점입니다. 기본은 Sites로 열어두고, 필요한 부분만 Forms, Sheets, Apps Script로 확장하면 교사가 훨씬 자유롭게 수업용 웹페이지를 구성할 수 있습니다.
실행 가능 체크: Google Sites는 자료 허브 제작에는 가장 안정적입니다. 초보자는 처음부터 Apps Script까지 붙이기보다 Sites, Drive, Forms 권한이 제대로 열리는지 먼저 확인하세요. 그 뒤 자동화 기능을 추가하는 순서가 좋습니다.

Tally의 백엔드 역할과 한계

백엔드처럼 보이는 부분

학생이 폼을 제출하면 Tally가 응답을 저장하고 교사에게 결과를 보여줍니다. 이 의미에서는 입력 수집형 백엔드처럼 작동합니다.

잘 맞는 범위

신청, 피드백, 익명 의견, 사전 조사처럼 “학생이 한 번 제출하고 교사가 확인하는” 활동에 잘 맞습니다.

한계

학생별 진행률, 로그인 기반 활동, 복잡한 조건 처리, 교사용 맞춤 대시보드는 직접 만든 백엔드보다 자유도가 낮습니다.

다음 단계

응답을 계속 관리해야 한다면 Google Sheets, AppSheet, Supabase와 비교합니다. 단순 수집이면 Tally나 Google Forms가 더 쉽습니다.

주의할 점

응답 데이터가 어디에 저장되는지, 누가 볼 수 있는지, 학생 개인정보를 얼마나 받을지 먼저 정해야 합니다.

초보자 판단

“응답을 받기만 하면 된다”면 Tally, “받은 응답을 상태별로 관리해야 한다”면 AppSheet나 시트 기반 관리가 더 적합합니다.

실제 진행 순서

  1. 먼저 받을 정보와 받지 않을 정보를 나눕니다. 이름이 꼭 필요하지 않다면 별칭, 모둠 번호, 선택지만 받습니다.
  2. Tally에서 새 폼을 만들고 제목과 설명을 적습니다. 설명에는 응답 목적과 소요 시간을 적습니다.
  3. 첫 질문은 부담이 적은 객관식으로 시작합니다. 학생이 바로 응답 흐름에 들어오게 하는 것이 좋습니다.
  4. 단답형, 객관식, 체크박스, 긴 글 응답 중 질문 목적에 맞는 형식을 고릅니다.
  5. 필수 질문과 선택 질문을 구분합니다. 모든 문항을 필수로 만들면 학생이 중간에 포기할 수 있습니다.
  6. 질문이 많다면 조건부 질문을 사용해 학생이 선택한 내용에 따라 필요한 문항만 보이게 합니다.
  7. 제출 후 메시지에는 “응답이 제출되었습니다”만 쓰지 말고 다음 활동 링크나 안내 문구를 넣습니다.
  8. 교사 계정에서 응답 결과가 어디에 모이는지 확인하고, 필요한 경우 내려받기나 연동 방법을 살펴봅니다.
  9. 학생용 링크를 복사하기 전에 비공개 정보, 이메일 수집 여부, 중복 제출 허용 여부를 확인합니다.
  10. Canva, Google Sites, HTML 페이지의 버튼에 Tally 링크를 연결합니다.
  11. 학생 기기에서 폼을 열어보고 글자가 잘 보이는지, 제출 후 안내가 맞게 나오는지 테스트합니다.
  12. 수업 후에는 응답 결과를 그대로 공개하지 말고 개인을 추정할 수 있는 서술형 응답이 있는지 먼저 확인합니다.
단순 설문은 Google Forms가 가장 익숙합니다. 다만 학생이 모바일에서 보기 편한 신청·피드백 화면이 필요하면 Tally도 좋은 선택지입니다.

초보자 판단

실제 진행 순서

  1. 먼저 앱으로 관리할 대상을 정합니다. 예: 제출 현황, 모둠 활동 체크, 답사 준비물, 기자재 대여
  2. Google Sheets에 기본 표를 만듭니다. 첫 행에는 열 이름을 넣고, 열 이름은 짧고 분명하게 씁니다.
  3. 학생 개인정보를 줄이기 위해 `이름` 대신 `별칭`, `모둠`, `번호`처럼 필요한 최소 항목을 검토합니다.
  4. 열 예시를 정합니다. 예: `모둠`, `활동주제`, `자료제출`, `발표준비`, `교사확인`, `메모`
  5. 표에 샘플 데이터 2~3개를 넣습니다. 빈 표만 연결하면 앱 화면을 상상하기 어렵습니다.
  6. AppSheet에서 새 앱을 만들고 해당 Google Sheets 파일을 연결합니다.
  7. 자동 생성된 입력 화면, 목록 화면, 상세 화면을 하나씩 눌러봅니다.
  8. 학생이 입력할 항목과 교사만 볼 항목을 구분합니다. 교사 확인 메모는 학생에게 열 필요가 없을 수 있습니다.
  9. 입력값 형식을 정합니다. 제출 여부는 Yes/No, 날짜는 Date, 모둠은 선택 목록처럼 설정하면 실수가 줄어듭니다.
  10. 테스트 계정으로 열어 보고 수정, 삭제, 조회 권한이 적절한지 확인합니다.
  11. 학생에게 공개할 앱인지, 교사만 사용할 앱인지 먼저 결정하고 공유 범위를 설정합니다.
  12. 필요하면 Google Sites나 수업 웹페이지에서 앱 링크로 연결합니다.
  13. 실제 수업 전 샘플 데이터를 지우거나 별도 테스트용 시트로 분리합니다.
AppSheet는 “웹페이지를 예쁘게 만드는 도구”라기보다 “표를 앱처럼 쓰게 만드는 도구”입니다. 제출 현황과 활동 체크처럼 교사가 계속 확인해야 하는 자료에 잘 맞습니다.
실행 가능 체크: AppSheet는 초보자도 시트가 준비되어 있으면 시작할 수 있습니다. 다만 처음부터 학생 전체에게 공개하기보다 교사용 내부 관리 앱으로 먼저 테스트하세요. 권한 설정을 확인한 뒤 학생 입력 화면을 여는 순서가 안전합니다.

Netlify의 백엔드 기능과 한계

기본 역할

Netlify는 기본적으로 HTML, CSS, JavaScript 파일을 웹주소로 공개하는 배포 도구입니다. 파일만 올리는 단계에서는 백엔드가 아닙니다.

백엔드처럼 쓰는 경우

Netlify Forms나 Functions 같은 기능을 쓰면 폼 제출 처리, 간단한 API, 외부 서비스 연결을 할 수 있습니다.

잘 맞는 범위

간단한 문의 폼, 외부 API 호출, 비밀 키를 브라우저 코드에 직접 노출하지 않는 중간 처리에 활용할 수 있습니다.

한계

처음 배포를 배우는 단계에서는 Functions 설정이 어렵게 느껴질 수 있습니다. 학생 응답 장기 관리는 Supabase나 Firebase가 더 명확할 수 있습니다.

주의할 점

비밀 키, 학생 개인정보, 외부 API 연결이 들어가면 단순 배포와 달리 보안 점검이 필요합니다.

초보자 판단

첫 배포는 드래그앤드롭으로 끝내고, 저장·처리가 필요해질 때 Forms나 Functions를 별도 단계로 검토합니다.

실제 진행 순서

  1. `class-page` 같은 새 폴더를 만들고 그 안에 `index.html`을 넣습니다.
  2. 이미지, PDF, CSS, JavaScript 파일이 있다면 같은 폴더 안에 정리합니다. 파일 이름은 가능하면 영어와 숫자로 둡니다.
  3. HTML 코드 안의 이미지 경로와 실제 파일 위치가 맞는지 확인합니다. 예: `assets/photo.png`라면 assets 폴더 안에 photo.png가 있어야 합니다.
  4. 브라우저에서 `index.html`을 먼저 열어 화면, 버튼, 링크, 모바일 크기를 확인합니다.
  5. 학생 개인정보나 학교 내부 자료가 HTML 파일에 직접 들어가 있지 않은지 확인합니다.
  6. Netlify에 로그인하고 Add new site 또는 Deploy manually를 선택합니다.
  7. `index.html` 파일만 끌어다 놓지 말고, 이미지와 CSS가 들어 있는 폴더 전체를 끌어다 놓습니다.
  8. 배포가 끝나면 생성된 주소를 눌러 실제 웹주소에서 화면을 확인합니다.
  9. 이미지가 깨지면 파일명 대소문자, 한글 파일명, 폴더 위치를 먼저 확인합니다.
  10. 주소 이름을 바꾸고 싶으면 Site settings에서 site name을 수정합니다.

GitHub Pages 실제 진행 순서

아래 순서는 `index.html` 파일이 이미 준비되어 있다는 가정에서 출발합니다. 처음에는 GitHub를 “온라인 파일함”으로, Pages를 “그 파일함에 놓인 안내판을 공개 주소로 바꾸는 기능”으로 이해하면 됩니다.

시작 전 준비물

  1. `index.html` 파일을 준비합니다. 파일 이름이 `index`가 아니라 정확히 `index.html`인지 확인합니다.
  2. 이미지, CSS, JavaScript 파일이 있다면 같은 폴더 안에 함께 둡니다. 처음에는 파일 이름을 영어와 숫자로 만드는 것이 안전합니다.
  3. 브라우저에서 `index.html`을 직접 열어 화면이 보이는지 먼저 확인합니다. 내 컴퓨터에서 안 열리는 파일은 인터넷에서도 잘 열리지 않습니다.
  4. 학생 이름, 학번, 성적, 얼굴 사진, 학교 내부 문서 링크가 파일 안에 들어 있지 않은지 확인합니다.
  5. GitHub 가입에 사용할 이메일을 준비합니다. 나중에 확인 코드가 이메일로 올 수 있습니다.

1단계: GitHub 가입하기

  1. 인터넷 주소창에 `github.com`을 입력해 GitHub에 접속합니다.
  2. 오른쪽 위 또는 첫 화면에 보이는 `Sign up` 버튼을 누릅니다. 화면 구성은 바뀔 수 있지만 가입 버튼은 보통 Sign up이라고 표시됩니다.
  3. 자주 쓰는 이메일 주소를 입력합니다. 학교 계정을 사용할 경우 학교 정책상 외부 서비스 가입이 가능한지 확인합니다.
  4. 비밀번호를 만듭니다. 다른 사이트와 같은 비밀번호를 쓰지 않는 것이 좋습니다.
  5. 사용자 이름을 영어로 정합니다. 이 이름은 나중에 GitHub 주소 일부가 될 수 있으므로 너무 복잡하지 않게 만듭니다.
  6. 이메일 확인 코드가 오면 메일함에서 코드를 확인해 입력합니다.
  7. 가입이 끝나면 GitHub에 로그인된 상태인지 확인합니다.

2단계: Repository 만들기

  1. GitHub 화면에서 `New` 또는 `New repository` 버튼을 찾습니다. 저장소를 새로 만드는 버튼입니다.
  2. `Repository name` 칸에 영어 이름을 적습니다. 예: `class-guide-2026`, `geo-project-page`, `my-first-page`
  3. 설명 칸은 비워도 됩니다. 쓰고 싶다면 “지리 수행평가 안내 페이지”처럼 간단히 적습니다.
  4. `Public`을 선택합니다. GitHub Pages로 무료 공개하려면 처음에는 Public이 가장 이해하기 쉽습니다.
  5. `Private`은 비공개 보관함입니다. 학생에게 공개할 웹페이지를 만들 때는 공개 설정과 Pages 지원 조건을 따로 확인해야 합니다.
  6. 다른 선택지는 처음에는 건드리지 않아도 됩니다. `Create repository` 버튼을 누릅니다.
  7. 빈 저장소 화면이 나오면 성공입니다. 아직 섬만 생겼고, 파일은 올라가지 않은 상태입니다.

3단계: index.html 업로드하기

  1. 빈 저장소 화면에서 `uploading an existing file` 또는 `Add file → Upload files`를 찾습니다.
  2. 업로드 화면이 열리면 `index.html` 파일을 끌어다 놓습니다.
  3. 이미지, CSS, JavaScript가 있다면 관련 폴더도 함께 올립니다. `index.html`만 올리면 디자인이나 이미지가 깨질 수 있습니다.
  4. 업로드 목록에 파일이 보이는지 확인합니다. 파일 이름에 한글, 공백, 특수기호가 많으면 나중에 경로 문제가 생길 수 있습니다.
  5. 아래쪽의 `Commit changes` 버튼을 누릅니다. 이것은 “파일 올리기를 저장합니다”라는 뜻입니다.
  6. 저장소 첫 화면으로 돌아와 `index.html`이 파일 목록에 보이면 업로드가 된 것입니다.
Public 저장소 위험 예시 Public 저장소는 온라인에 공개된 자료함에 가깝습니다. 학생 명단, 학번, 성적표, 상담 내용, 비공개 Drive 링크, 학교 내부 문서, API 비밀 키는 올리지 않습니다. 수업 안내문처럼 공개되어도 되는 자료만 올리고, 학생 자료는 익명 예시로 바꾼 뒤 테스트하세요.

4단계: GitHub Pages 켜기

  1. 저장소 화면 위쪽 메뉴에서 `Settings`를 누릅니다.
  2. 왼쪽 메뉴에서 `Pages`를 찾습니다. 메뉴가 많으면 아래로 천천히 내려봅니다.
  3. `Build and deployment` 영역을 찾습니다.
  4. `Source`가 보이면 `Deploy from a branch`를 선택합니다. 이미 선택되어 있으면 그대로 둡니다.
  5. `Branch`에서 `main`을 선택합니다. 예전 저장소는 `master`라고 보일 수도 있습니다.
  6. 폴더는 `/root`를 선택합니다. `index.html`을 저장소 맨 위에 올렸다면 `/root`가 가장 일반적입니다.
  7. `Save`를 누릅니다. 이제 GitHub가 웹페이지 다리를 놓기 시작합니다.
  8. 바로 주소가 안 보여도 1~2분 정도 기다립니다. 새로고침하면 `Your site is live at ...` 같은 안내가 나타날 수 있습니다.
  9. 생성된 주소를 눌러 웹페이지가 열리는지 확인합니다.

5단계: 안 열릴 때 확인하기

  1. 404가 나오면 `index.html`이 저장소 맨 위에 있는지 확인합니다. 폴더 안에 들어가 있으면 Pages가 첫 파일을 못 찾을 수 있습니다.
  2. 파일 이름이 `Index.html`, `index.htm`, `index.html.txt`처럼 되어 있지 않은지 확인합니다.
  3. 이미지가 깨지면 이미지 파일도 함께 올렸는지, HTML에 적힌 파일 이름과 실제 파일 이름의 대소문자가 같은지 확인합니다.
  4. Pages 주소가 안 생기면 Settings → Pages에서 Branch가 `main`, 폴더가 `/root`로 저장되어 있는지 다시 봅니다.
  5. 수정했는데 반영이 늦으면 몇 분 기다린 뒤 새로고침합니다. 브라우저가 이전 화면을 기억하고 있을 수 있습니다.
  6. 학생에게 공유하기 전 휴대폰이나 아이패드에서 주소를 열어보고, 버튼과 링크가 잘 눌리는지 확인합니다.
GitHub Pages는 처음엔 낯설지만, “수정 기록이 남는 수업 자료실”을 만들 수 있다는 점이 강점입니다.
실행 가능 체크: GitHub Pages는 초보자도 가능하지만 Netlify보다 용어가 많습니다. `저장소는 온라인 폴더`, `Commit은 저장`, `Pages는 공개 주소 만들기`로 바꿔 설명하면 따라 하기 쉬워집니다.
관련 사례: 프로그램 안내 페이지처럼 매년 수정해 다시 쓸 자료, 기후 탐색 웹앱처럼 파일을 계속 고쳐가며 운영할 자료에 적합합니다.

Cloudflare의 백엔드 기능과 한계

기본 역할

Cloudflare Pages는 기본적으로 정적 사이트와 프론트엔드 프로젝트를 배포하는 도구입니다. Pages만 쓰면 백엔드가 아닙니다.

백엔드처럼 쓰는 경우

Workers, Pages Functions, D1, KV, R2 같은 기능을 함께 쓰면 API, 간단 데이터 저장, 파일 저장, 서버 처리 역할을 만들 수 있습니다.

잘 맞는 범위

장기 운영하는 웹앱, 빠른 응답이 필요한 API, 정적 사이트에 작은 서버 기능을 붙이는 경우에 적합합니다.

한계

초보자에게는 Pages, Workers, D1, KV의 차이가 어렵습니다. HTML 첫 배포 단계라면 Netlify나 GitHub Pages가 더 쉽습니다.

주의할 점

Workers나 데이터 저장 기능을 쓰면 공개 범위, 환경 변수, 접근 권한, 로그에 남는 데이터까지 확인해야 합니다.

초보자 판단

Cloudflare는 “GitHub 자동 배포에 익숙해진 뒤, 서버 기능도 붙이고 싶을 때” 검토하는 운영형 선택지로 설명하는 것이 좋습니다.

처음이라면 건너뛰어도 됩니다 Cloudflare Pages는 좋은 운영 도구지만, 첫 웹페이지 배포를 위한 필수 단계는 아닙니다. GitHub 저장소, 빌드 설정, 미리보기 배포, 환경 변수라는 말이 낯설다면 Netlify나 Google Sites로 먼저 성공 경험을 만드는 편이 좋습니다. Workers나 Functions를 쓸 때 비밀 키가 로그나 공개 코드에 남지 않도록 반드시 확인해야 합니다.

Cloudflare Pages 실제 진행 순서

  1. GitHub 저장소에 `index.html`이 올라가 있는지 확인합니다. 저장소 첫 화면에서 `index.html`이 보이면 가장 쉽습니다.
  2. Cloudflare에 가입하거나 로그인합니다.
  3. Cloudflare 대시보드에서 `Workers & Pages` 또는 `Pages` 메뉴를 찾습니다. 화면 이름은 바뀔 수 있으니 Pages라는 단어를 기준으로 찾습니다.
  4. `Create application`, `Create project`, `Pages` 중 새 Pages 프로젝트를 만드는 버튼을 선택합니다.
  5. `Connect to Git` 또는 `Import an existing Git repository`를 선택합니다. 이것은 GitHub에 있는 저장소를 Cloudflare와 연결하겠다는 뜻입니다.
  6. GitHub 로그인을 요청하면 연결을 허용합니다. 이때 Cloudflare가 어떤 저장소에 접근할 수 있는지 권한 범위를 확인합니다.
  7. 배포할 GitHub 저장소를 선택합니다. 예: `class-guide-2026`
  8. 프로젝트 이름은 보통 저장소 이름과 비슷하게 둡니다. 이 이름이 `pages.dev` 주소 일부가 될 수 있습니다.
  9. `Production branch`는 보통 `main`을 선택합니다. 예전 저장소라면 `master`가 보일 수도 있습니다.
  10. 정적 HTML 한 장짜리 사이트라면 프레임워크 선택은 `None` 또는 framework 없음에 가깝게 둡니다.
  11. 빌드가 필요 없는 정적 HTML은 먼저 `Build command`를 비워두는 방향으로 확인합니다. 화면에서 값을 꼭 요구하면 공식 문서나 AI에게 현재 화면을 보여주고 물어본 뒤 입력합니다.
  12. `Build output directory`는 웹사이트 파일이 실제로 들어 있는 위치입니다. `index.html`이 저장소 맨 위에 있다면 루트 위치를 의미하도록 설정합니다.
  13. 화면에서 output directory 값을 어떻게 넣어야 할지 헷갈리면, 현재 화면의 항목 이름과 저장소 파일 구조를 AI에게 보여주고 물어봅니다.
  14. `Save and Deploy` 또는 `Deploy`를 누릅니다.
  15. 배포 로그가 지나간 뒤 `*.pages.dev` 형태의 주소가 생기면 눌러서 확인합니다.
  16. 404가 나오면 `index.html`이 최상위 위치에 있는지, output directory가 실제 파일 위치를 가리키는지 확인합니다.
  17. 이후 GitHub에서 파일을 수정하고 commit하면 Cloudflare Pages가 자동으로 새 배포를 시작하는지 확인합니다.
  18. 학생에게 공유하기 전 실제 공개 주소와 미리보기 주소를 구분합니다. 수업에는 안정적인 공개 주소를 공유하는 것이 좋습니다.

Build command

웹사이트를 공개용 파일로 포장할 때 실행하는 명령입니다. 단순 `index.html`은 포장 과정이 없으므로 보통 비워두면 됩니다. 무엇을 넣어야 할지 모르겠다면 임의로 입력하지 말고, 현재 화면과 파일 구조를 AI에게 보여주고 확인하세요.

Output directory

테스트 모드 공개 위험 예시 테스트 모드나 느슨한 보안 규칙을 그대로 두면, 학생이 보지 않아야 할 데이터가 읽히거나 수정될 수 있습니다. 파일 업로드를 열어두면 학생 얼굴 사진, 저작권 자료, 큰 용량 파일이 그대로 저장될 수 있습니다. Firebase는 강력하지만 “로그인별 권한을 설명할 수 있을 때” 수업에 적용하는 것이 안전합니다.

실제 진행 순서

  1. 먼저 Firebase가 꼭 필요한지 확인합니다. 단순 안내, 설문, 신청은 Canva, Google Sites, Forms, Tally가 더 쉽습니다.
  2. 실시간 데이터, 로그인, 파일 업로드, 여러 기능이 있는 웹앱이 필요할 때 Firebase를 검토합니다.
  3. 저장할 데이터와 공개하면 안 되는 데이터를 나눕니다. 예: 퀴즈 점수는 저장하더라도 이름 대신 별칭을 쓸 수 있습니다.
  4. Firebase Console에서 새 프로젝트를 만듭니다.
  5. 웹앱을 추가하고 설정 코드를 확인합니다.
  6. 데이터 저장이 필요하면 Firestore Database를 만듭니다.
  7. 파일 업로드가 필요하면 Storage 사용 여부를 따로 검토합니다. 파일 저장은 개인정보 위험이 커질 수 있습니다.
  8. HTML/JavaScript 또는 React 코드에 Firebase 설정을 연결합니다.
  9. 테스트 데이터를 저장하고 Firebase Console에서 실제로 저장되는지 확인합니다.
  10. 로그인을 붙인다면 학생이 어떤 계정으로 들어오는지, 교사가 어떤 데이터를 볼 수 있는지 정합니다.
  11. 보안 규칙을 반드시 점검합니다. 테스트 모드 그대로 공개하지 않습니다.
  12. 호스팅까지 Firebase에서 할지, 배포는 Vercel/Netlify에 맡길지 결정합니다.
  13. 학생 기기에서 테스트하고, 수업 후 데이터 삭제나 보관 기준을 정합니다.
실행 가능 체크: Firebase는 기능 확장성은 크지만 초보자에게는 설정이 많습니다. 실시간 동기화, 로그인, 파일 업로드가 꼭 필요한 경우에만 선택하세요. 공개 전 보안 규칙을 AI에게 다시 점검받는 절차를 넣는 것이 안전합니다.
관련 사례: 자료 탐색 플랫폼이 로그인 기반 학습 기록이나 실시간 상태 표시까지 확장될 때 검토합니다. 단순 설문이나 안내 페이지에는 과한 선택일 수 있습니다.

어떤 도구를 백엔드처럼 볼 수 있을까요?

백엔드는 학생이 보는 화면 뒤에서 저장, 조회, 권한, 자동 처리를 맡는 역할입니다. 모든 도구가 같은 수준의 백엔드는 아니므로, 수업 목적과 위험도를 기준으로 골라야 합니다.

Canva SheetsCanva 내부 표 저장·시각화형입니다.교사가 직접 관리하는 모둠 현황, 사전 조사 요약, 발표 순서, 활동 체크를 표와 차트로 만들기 좋습니다.학생별 로그인, 행별 권한, 외부 웹앱 API 저장이 필요한 본격 백엔드와는 구분해야 합니다.
Google Forms·Tally입력 수집형입니다.학생 응답을 받아 저장해주지만, 직접 만든 웹앱처럼 자유로운 조회·수정·권한 구조를 만들기는 어렵습니다.
Apps ScriptGoogle 기반 가벼운 백엔드입니다.Sheets 저장, 제출 조회, 자동 이메일, 간단 웹앱을 만들 수 있습니다.다만 실행 권한과 사용량 제한, 공개 범위 확인이 필요합니다.
AppSheet시트 기반 관리 앱입니다.Google Sheets를 앱처럼 입력·조회·관리하게 해줍니다.공개용 디자인 웹페이지나 복잡한 게임형 활동에는 맞지 않을 수 있습니다.
Netlify기본은 배포 도구입니다.Forms나 Functions를 붙이면 간단한 서버 처리를 할 수 있습니다.첫 배포 단계에서는 정적 사이트 공개용으로 보는 것이 쉽습니다.
Cloudflare운영형 배포+서버 기능입니다.Pages, Workers, D1, KV, R2를 조합할 수 있습니다.초보자에게는 설정 개념이 많습니다.
Vercel프로젝트형 웹앱 배포+서버 함수입니다.React/Next.js 기반 웹앱에서 API 처리까지 할 수 있습니다.HTML 한 장짜리 안내 페이지에는 과할 수 있습니다.
Supabase표 기반 백엔드입니다.제출 목록, 의견, 지도 데이터처럼 행과 열로 관리할 데이터에 좋습니다.RLS와 API key 개념을 확인해야 합니다.
Firebase실시간·로그인 중심 백엔드입니다.실시간 퀴즈, 로그인 기반 기록, 파일 저장에 강합니다.단순 설문에는 과하고 보안 규칙 점검이 필수입니다.
