수업자료 제작 방식 이해하기

이 영역은 특정 프로그램 사용법이 아닙니다. 어떤 수업자료를 만들지 판단하는 지도입니다. 만들고 싶은 결과를 먼저 정한 뒤 프로그램별 상세 가이드에서 실제 진행 순서를 확인하면 됩니다.

먼저 알아둘 점: 화면과 버튼 이름은 바뀔 수 있습니다 GitHub, Netlify, Cloudflare, Vercel은 메뉴 이름과 버튼 위치가 조금씩 바뀔 수 있습니다. 이 안내서는 초보자가 흐름을 이해하고 따라가도록 만든 길잡이입니다. 실제 화면이 다르면 공식 도움말이나 AI 검색 창구를 활용하세요. 그때는 “현재 보이는 버튼 이름”과 “막힌 화면”을 함께 설명하는 것이 좋습니다.

목적별 제작 경로 보기

먼저 판단하기: 웹페이지가 꼭 필요한가?

웹페이지 제작은 수업을 돕기 위한 선택지입니다. 단순 공지, 과제 제출, 파일 공유만 필요하다면 Google Classroom, Padlet, Google Forms가 더 빠를 수 있습니다. 여러 자료, 활동, 제출 안내를 한 흐름으로 묶어야 할 때 웹페이지 제작을 검토합니다.

AI·노코드에서 바로 만드는 길

코드를 직접 만지지 않고 대화하거나 화면에서 조립해 수업자료를 만드는 방식입니다. Canva AI, Google Sites, Genially처럼 입력창에 설명하거나 블록을 배치합니다. 안내 페이지, 활동 화면, 자료 허브를 빠르게 만들 수 있습니다.

대체 가능: 단순 공지와 과제 흐름은 Classroom/LMS, 간단 카드뉴스는 Canva 디자인만으로 충분할 수 있습니다.

관련 사례: 세특 전략 특강 안내, 수업량 유연화 학생 안내

HTML 기반 정적 안내 페이지 길

`index.html`을 중심으로 만드는 방식입니다. 수업 안내, 수행평가 안내, 자료 링크 모음, 작품 전시에 잘 맞습니다. 움직임은 많지 않지만 주소 하나로 학생에게 수업 흐름을 보여주기 좋습니다.

대체 가능: 파일 공유와 안내만 목적이라면 Google Sites, Notion, Drive 폴더, Classroom 게시글이 더 빠를 수 있습니다.

관련 사례: 수업량 유연화 프로그램 안내

상호작용 활동을 넣는 길

학생이 누르고, 고르고, 뒤집고, 탐색하는 활동이 필요한 경우입니다. OX 퀴즈, 플래시카드, 매칭 게임, 팝업 설명창, 지도 탐색처럼 “학생의 행동”이 학습을 돕는 자료를 만들 때 선택합니다.

대체 가능: 일반 퀴즈와 게임형 복습은 Kahoot, Quizizz, Wordwall, LearningApps가 더 빠를 수 있습니다.

관련 사례: 세계지리 기후 탐색 웹앱, 한국지리 기후 탐색 웹앱

제출·응답을 모으는 길

학생 의견, 신청 현황, 활동 완료 여부, 피드백을 받아야 할 때 선택합니다. 웹페이지 안에 모든 것을 저장하려 하기보다 Google Forms, Tally, AppSheet처럼 이미 안정적인 제출함을 연결하는 방식이 더 쉬운 경우가 많습니다.

관련 사례: 여행지리 학습자 분석 인포그래픽

웹앱처럼 확장하는 길

단순 안내문을 넘어서는 기능이 필요할 때입니다. 예: 여러 화면, 진행률, 필터, 대시보드, 지도 탐색 이 경우 AI가 만든 HTML/JavaScript나 React/Next.js 프로젝트를 검토합니다. 배포 도구도 함께 생각해야 합니다.

관련 사례: 진로진학 플랫폼

데이터를 저장하고 다시 불러오는 길

학생 응답을 웹페이지가 기억하고 다시 보여줘야 할 때입니다. 교실로 비유하면 학생 답변을 넣어두는 기록장을 따로 마련하는 일입니다. Supabase와 Firebase를 검토할 수 있지만, 공개 범위와 개인정보 최소화가 먼저입니다.

대체 가능: 응답을 받기만 하면 Forms/Tally, 교사용 현황판이면 Sheets/AppSheet, 가벼운 시각화면 Canva Sheets가 더 쉽습니다.

관련 사례: 여행지리 학습자 분석 인포그래픽

주소로 배포하고 공유하는 길

내 컴퓨터 안에 있는 웹페이지 파일을 학생들이 주소로 열 수 있게 인터넷에 올리는 방식입니다. 오늘 바로 공유하려면 Netlify를 검토합니다. 해마다 고쳐 쓸 자료라면 GitHub Pages를 검토합니다. 자동 반영과 장기 운영까지 생각한다면 Cloudflare Pages를 검토합니다.

관련 사례: 세계지리 기후 탐색 웹앱, 한국지리 기후 탐색 웹앱

이 페이지는 위에서 아래로 반드시 순서대로 따라가는 사다리가 아닙니다. “내 수업에는 어떤 길이 맞을까?”를 먼저 고르고, 필요한 길만 골라 프로그램별 상세 가이드로 넘어가면 됩니다.

Canva AI/웹페이지

HTML 파일 기반 제작 이해하기

이 부분은 파일을 직접 다루는 방식으로 갈 때만 읽으면 됩니다. Canva 웹페이지나 Google Sites처럼 자체 게시되는 도구만 쓸 때는 건너뛰어도 괜찮습니다.
index.html

학생이 주소를 열었을 때 가장 먼저 보는 첫 안내판입니다. 웹페이지 폴더 안에서 “여기부터 보여주세요”라고 알려주는 시작점입니다.

HTML

안내판에 무엇을 적을지 정하는 뼈대입니다. 제목, 설명 문장, 버튼, 링크처럼 학생이 볼 내용을 놓는 자리입니다.

CSS

안내판을 보기 좋게 꾸미는 규칙입니다. 글씨 크기, 색, 줄 간격, 카드 배치처럼 학생이 읽기 편한 모양을 정합니다.

JavaScript

안내판에 반응을 넣는 장치입니다. 버튼을 누르면 내용이 바뀌거나, 퀴즈 정답이 보이거나, 카드를 뒤집는 움직임을 담당합니다.

assets 폴더

이미지, 아이콘, PDF, 활동지처럼 웹페이지와 함께 움직여야 하는 자료 보관함입니다. 안내판만 올리고 자료함을 빼먹으면 학생 화면에서 그림이나 파일이 보이지 않을 수 있습니다.

한 파일로 만들기

처음에는 내용, 꾸미기, 반응을 `index.html` 한 장 안에 모두 넣는 방식이 쉽습니다. 챙길 파일이 하나라서 배포 실수가 줄어듭니다.

여러 파일로 나누기

사이트가 커지면 꾸미기 규칙, 반응 장치, 이미지 자료함을 따로 나눕니다. 정리는 쉬워지지만 배포할 때 폴더 구조를 그대로 챙겨야 합니다.

배포 전 확인

Netlify, GitHub Pages, Cloudflare Pages에 올릴 때는 `index.html`만이 아니라 이미지와 자료 폴더까지 같은 구조로 올라갔는지 확인해야 합니다.

내 컴퓨터에 웹페이지 폴더 만들기

1. 폴더 만들기

바탕화면이나 문서 폴더에 `class-webpage`처럼 영어 이름의 새 폴더를 만듭니다. 이 폴더가 수업 안내판을 담는 작은 서랍입니다.

2. index.html 저장하기

AI가 준 코드를 새 파일에 붙여 넣습니다. 파일 이름은 반드시 `index.html`로 저장합니다. 주소를 열었을 때 가장 먼저 찾는 첫 안내판입니다.

3. 확장자 확인하기

파일 이름이 `index.html.txt`가 되면 웹페이지로 인식되지 않을 수 있습니다. 파일 이름 끝이 정확히 `.html`인지 확인합니다.

4. 자료 폴더 챙기기

이미지, PDF, 활동지 파일이 있으면 `images`나 `assets` 폴더에 넣고, 배포할 때 그 폴더까지 함께 올립니다.

5. 먼저 열어보기

`index.html`을 더블클릭해 브라우저에서 열어봅니다. 내 컴퓨터에서 글자, 버튼, 이미지가 보이는지 먼저 확인합니다.

6. 폴더 전체 배포하기

Netlify에 올릴 때는 파일 하나만이 아니라 `index.html`과 자료 폴더가 들어 있는 폴더 전체를 올립니다.

HTML 실험형 첫 성공 체크리스트 아래 다섯 가지가 모두 되면 첫 배포 실습은 성공으로 봐도 됩니다. 완벽한 디자인보다 “학생이 열고, 읽고, 누르고, 제출 링크로 이동할 수 있는가”가 먼저입니다.
1. 첫 파일폴더 안에 `index.html`이 있습니다.파일 이름이 `index.html.txt`가 아닌지 확인합니다.
2. 자료 폴더이미지, PDF, 활동지 파일이 있다면 같은 폴더 안에 들어 있습니다.Netlify에 올릴 때도 이 폴더를 함께 올립니다.
3. 내 컴퓨터 확인`index.html`을 더블클릭했을 때 글자, 버튼, 이미지가 보입니다.가장 중요한 제출 링크나 자료 링크를 직접 눌러봅니다.
4. 웹주소 확인Netlify 같은 배포 도구에 폴더 전체를 올린 뒤 웹주소가 생깁니다.주소를 새 브라우저 창에서 열었을 때 같은 화면이 보입니다.
5. 학생 기기 확인교사용 컴퓨터가 아니라 학생 휴대폰이나 학생 계정에서 열어봅니다.글자가 너무 작거나 버튼이 너무 붙어 있으면 공개 전에 다시 수정합니다.
학생 화면 확인은 따로 점검하세요 배포 주소가 생겼다면 끝이 아닙니다. 교사 휴대폰, 학생 계정, 시크릿 창에서 열어보는 방법은 공개 전 점검의 학생 기기 확인에서 이어서 확인하세요.

먼저 열어볼 수 있는 시작 템플릿 빈 파일에서 시작하기 어렵다면 사례 보기 페이지의 템플릿을 먼저 열어보세요. 안내 페이지, 퀴즈 활동, 응답 연결 허브처럼 목적별 구조를 눈으로 확인합니다. 그다음 AI에게 “이 구조를 내 수업 주제로 바꿔줘”라고 요청하면 훨씬 덜 막힙니다.

AI로 첫 index.html 만들기 순서

  1. 먼저 수업 목적을 한 문장으로 씁니다. 예: “도시 구조 단원 수행평가 안내 페이지를 만들고 싶다.”
  2. 대상 학년, 수업 시간, 학생이 해야 할 행동, 넣을 링크를 적습니다.
  3. 처음에는 “HTML, CSS, JavaScript를 한 파일 안에 넣어줘”라고 요청합니다.
  4. AI가 만든 코드를 `index.html`이라는 이름으로 저장합니다.
  5. 파일을 브라우저로 열어 글자, 버튼, 링크, 모바일 화면을 확인합니다.
  6. 마음에 들지 않는 부분은 “버튼을 더 크게”, “문장을 더 짧게”, “모바일에서 카드가 한 줄로 보이게”처럼 다시 요청합니다.
  7. 공개 전에 학생 이름, 학번, 성적, 얼굴 사진, 학교 내부 링크가 들어가지 않았는지 확인합니다.

AI에게 바로 넣을 수 있는 첫 프롬프트

나는 교사야.
수업용 웹페이지를 처음 만들어보려고 해.

목표:
- 고등학교 지리 수행평가 안내 웹페이지 만들기

조건:
- 파일 이름은 index.html
- HTML, CSS, JavaScript를 한 파일 안에 넣어줘
- 모바일에서도 잘 보이게 반응형으로 만들어줘
- 섹션은 수업 목표, 활동 순서, 제출 링크, 평가 기준, FAQ로 구성해줘
- Google Forms 제출 버튼을 넣을 자리를 만들어줘
- 학생 개인정보를 직접 입력하지 않도록 주의 문구를 넣어줘
- 초보자인 내가 그대로 저장해서 열 수 있게 전체 코드를 제공해줘

디자인:
- 교사용 안내서처럼 차분하고 읽기 쉽게
- 버튼은 크고 명확하게
- 학생이 다음 행동을 바로 알 수 있게

API와 데이터 저장 이해하기

프론트엔드

학생이 보는 화면입니다. 안내문, 버튼, 퀴즈, 지도처럼 눈에 보이는 부분입니다.

백엔드

학생 화면 뒤에서 자료를 저장하고 불러오는 공간입니다. 교무실 안 기록 보관함처럼 이해하면 됩니다.

데이터베이스

학생 응답, 제출 시간, 활동 완료 여부를 보관하는 기록장입니다. Supabase는 표처럼, Firebase는 실시간으로 바뀌는 온라인 기록판처럼 이해하면 쉽습니다.

API

학생 화면과 기록 보관함이 서로 자료를 주고받는 통로입니다. 접수 창구와 교무실 서랍 사이의 전달 통로로 볼 수 있습니다.

API key

그 통로를 사용할 수 있게 해주는 출입 카드입니다. 공개 웹페이지에 들어갈 수 있는 키도 있지만, 교사용 마스터키처럼 절대 공개하면 안 되는 비밀 키도 있습니다.

보안 규칙

누가 어떤 기록을 볼 수 있고, 쓸 수 있고, 고칠 수 있는지 정하는 잠금장치입니다. 학생 개인정보가 있으면 반드시 확인해야 합니다.

낯선 영어가 나오면 이렇게 바꿔 읽으세요 GitHub, Netlify, Cloudflare 같은 서비스는 영어 메뉴가 많아서 초보자가 여기서 가장 자주 멈춥니다. 아래 뜻만 알고 있어도 “내가 지금 무엇을 정하는 단계인지”를 훨씬 쉽게 따라갈 수 있습니다.
Repository온라인 파일함 또는 수업자료 보관함입니다.`index.html`, 이미지, CSS 파일을 이 보관함에 넣는다고 생각하면 됩니다.
Commit“이 상태로 저장합니다”라는 저장 도장입니다.GitHub에서 파일을 올리거나 고친 뒤에는 commit을 눌러야 실제로 반영됩니다.
Branch어떤 파일 묶음을 공개 기준으로 삼을지 정하는 선택지입니다.처음에는 보통 `main`을 고르면 됩니다. 예전 자료는 `master`라고 보일 수도 있습니다.
Deploy내 컴퓨터나 저장소 안의 파일을 학생이 열 수 있는 웹주소로 올리는 일입니다.게시, 배포, 공개 주소 만들기라고 이해하면 됩니다.
Build command웹앱을 공개용 파일로 포장할 때 쓰는 명령입니다.단순 `index.html` 사이트는 보통 빌드 명령이 필요 없습니다.화면에서 요구하면 현재 파일 구조를 AI나 공식 문서로 다시 확인하세요.
Output directory서비스가 실제로 공개할 파일이 들어 있는 위치입니다.처음에는 `index.html`이 어디에 있는지 묻는 칸이라고 생각하면 됩니다.
API key화면과 저장 공간이 대화할 때 쓰는 출입 카드입니다.공개해도 되는 키와 절대 공개하면 안 되는 관리자 키가 있으므로, 학생이 보는 HTML에는 비밀 키를 넣지 않습니다.
현재 방법을 확인할 수 있는 공식 창구 메뉴 이름이 다를 때는 공식 문서에서 최신 흐름을 확인하세요. GitHub 저장소 만들기, GitHub 파일 추가, GitHub Pages 게시 설정, Netlify 배포, Cloudflare Pages Git 연결을 참고할 수 있습니다.
API key를 볼 때의 안전 원칙 공개 웹페이지 코드에 보이는 키가 모두 비밀 키는 아닙니다. 하지만 Supabase의 service role key, 외부 유료 API의 비밀 키, 서버 전용 관리자 키는 학생이 보는 HTML 코드에 넣으면 안 됩니다. 공개 가능한 키를 쓰더라도 보안 규칙이 없으면 학생 응답이 노출될 수 있습니다. 실제 개인정보를 저장하기 전에는 반드시 테스트 데이터로 먼저 확인하세요.
다음에 볼 페이지 제작 흐름을 확인했다면, 이제 실제 선택과 실행으로 넘어가면 됩니다. 도구를 아직 못 골랐다면 도구 고르기를 확인하세요. AI에게 작업을 맡기고 싶다면 AI 활용, 작은 실습부터 해보고 싶다면 사례 기반 실습 워크북을 보면 됩니다. 프로그램별 실제 순서를 보고 싶다면 프로그램별 상세 가이드, 공개 전 안전 점검이 필요하면 공개 전 점검을 확인하세요.
