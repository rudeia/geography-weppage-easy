코딩을 몰라도 괜찮아요. 수업 안내, 자료 모음, 퀴즈 활동, 상호작용형 활동을 AI와 함께 작게 만들어볼 수 있습니다. 필요하다면 학생 응답 저장까지 확장할 수 있습니다. 중요한 것은 프로그램 이름이 아니라 선생님의 수업 아이디어입니다. 같은 도구도 활동 설계에 따라 전혀 다른 수업 도구로 발전할 수 있습니다.

처음 만드는 사람은 여기서 출발하세요

처음부터 모든 기능을 만들 필요는 없습니다. 오늘 성공할 수 있는 작은 목표를 고릅니다. 필요할 때만 다음 단계로 확장하면 됩니다.

완전 처음이라면 이 순서만 따라가도 됩니다 1. 먼저 “웹페이지가 정말 필요한 수업인가?”를 판단합니다. 2. 필요하다면 만들고 싶은 수업 장면을 고릅니다. 3. 제작 흐름에서 “안내, 활동, 응답, 배포” 중 어디에 해당하는지 확인합니다. 4. 실제로 만들 때는 AI 활용 또는 프로그램별 상세 가이드 중 하나만 봅니다. 5. 실습 워크북에서 30분, 60분, 90분 중 하나를 따라 해봅니다. 6. 학생에게 보내기 전 공개 전 점검을 합니다. 처음부터 모든 도구를 비교하지 않아도 됩니다. 오늘은 작은 결과물 하나를 학생 기기에서 열어보는 경험이 더 중요합니다.
필요성 먼저 판단하기 방식 고르기 실습 워크북 공개 전 점검
직접 만들기 전 확인 이미 Classroom, 학교 LMS, Forms, Padlet, Kahoot으로 수업이 잘 운영되고 있다면 새 웹페이지를 만들 필요가 없습니다. 이 안내서는 웹사이트 제작을 권하는 자료가 아닙니다. 기존 앱으로 부족할 때, 선생님이 더 자유롭게 선택하도록 돕는 자료입니다.
기존 앱으로 충분한지 5문항만 확인하세요 1. 학생 명단, 과제, 성적 흐름이 핵심인가요? 그러면 Classroom이나 학교 LMS가 먼저입니다. 2. 응답을 받기만 하면 되나요? 그러면 Google Forms나 Tally가 더 안전합니다. 3. 퀴즈 결과와 리포트가 필요하나요? 그러면 Kahoot, Quizizz, Wordwall을 먼저 검토하세요. 4. 자료와 링크만 모으면 되나요? 그러면 Google Sites, Notion, Drive, Classroom으로 충분할 수 있습니다. 5. 기존 앱으로는 수업 흐름이 끊기나요? 그때 직접 제작을 검토하면 됩니다.

  • 성공 기준: 새 제작 없이 기존 앱의 링크, 권한, 안내 문구만 정리됩니다.

프론트엔드만으로 가능할까요?

가능합니다. 지금처럼 첫 화면에서 다른 페이지로 이동하는 구조는 HTML, CSS, JavaScript만으로 만들 수 있습니다. 특정 페이지 안에서 프로그램별 탭을 보여주는 것도 프론트엔드만으로 가능합니다. 다만 학생 응답을 나중에 다시 확인하거나 학생별 기록을 나누어 관리하려면 별도 저장 도구가 필요합니다. 파일 업로드를 받는 기능도 처음에는 기존 폼 도구부터 검토하세요.

가능한 방식

  • 여러 HTML 파일로 구성하는 정적 멀티페이지 사이트
  • 한 HTML 안에서 JavaScript로 화면을 바꾸는 싱글페이지 방식
  • 탭, 필터, 버튼 반응은 JavaScript로 처리
  • Netlify, GitHub Pages, Cloudflare Pages에 그대로 배포 가능
  • 응답 저장은 Forms, Tally, AppSheet, Supabase, Firebase 같은 도구와 연결
